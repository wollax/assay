# M002: Real Assay Integration — Context

**Gathered:** 2026-03-17
**Status:** Ready for planning

## Project Description

Smelt is a Docker-first job runner that provisions containers, mounts the host repo, and delegates execution to Assay inside the container. M001 delivered the full container lifecycle with a mock Assay binary. M002 retires the mock and integrates the real Assay binary.

## Why This Milestone

M001's `AssayInvoker` (D029) was built against an assumed Assay manifest contract, tested only with mock shell scripts. Real Assay has been confirmed to have a structurally different manifest format — Smelt currently generates the wrong TOML. Every real `smelt run` that reaches the Assay exec phase will fail until M002 fixes this.

**Why now:** Docker plumbing works. The mock retirement is the highest-value next step — it turns Smelt from a container orchestrator that talks to a fake Assay into one that actually works.

## User-Visible Outcome

### When this milestone is complete, the user can:

- Run `smelt run manifest.toml` and have a real `assay` binary execute inside the container, creating worktrees and running quality gates
- See Assay's gate output streamed to the terminal as sessions complete
- Find the result branch on the host repo containing agent-authored commits, collected after real Assay execution

### Entry point / environment

- Entry point: `smelt run <manifest.toml>` (CLI)
- Environment: local dev with Docker daemon running + real `assay` binary installed in the target image or the container
- Live dependencies: Docker daemon, real `assay` binary, Claude API key (for Assay's agent sessions), git

## Completion Class

- Contract complete means: `AssayInvoker` generates a manifest and invocation that a real `assay run` binary accepts and executes without error — verified by unit tests against the real `assay-types` schema
- Integration complete means: `smelt run` with a real manifest provisions a container, installs/uses a real `assay` binary, generates valid Assay spec and manifest files, and `assay run` exits 0 with sessions recorded
- Operational complete means: result branch is collected from real Assay sessions, container is cleaned up, `smelt status` shows correct phase progression

## Final Integrated Acceptance

To call this milestone complete, we must prove:

- `smelt run manifest.toml` (non-mock) provisions a container, runs `assay run` with a real binary, and produces a result branch — demonstrated with a real Docker environment
- The Assay manifest and spec files generated by Smelt are valid per Assay's schema (`assay-types` serde structs with `deny_unknown_fields`)
- Gate output from inside the container streams to the terminal in real time

## The Assay Contract Gap (Critical)

This is the primary technical risk for M002. Research findings:

### What Smelt currently generates (wrong)

`AssayInvoker::build_manifest_toml()` produces:
```toml
[[session]]
name = "frontend"
spec = "Implement the login page..."
harness = "npm test -- --filter auth"
timeout = 300
```

### What Assay actually expects

Real Assay `RunManifest` (`assay-types::manifest`) uses `[[sessions]]` (plural) where `spec` is a **reference to an existing spec name** (not an inline description):
```toml
[[sessions]]
spec = "frontend"          # references .assay/specs/frontend.toml
name = "frontend-session"  # optional display name
depends_on = ["backend"]   # dependency ordering
```

Assay spec files (`GatesSpec` / `FeatureSpec`) in `.assay/specs/<name>.toml` contain:
```toml
name = "frontend"
description = "Implement the login page"

[gate]
enforcement = "required"

[[criteria]]
name = "tests"
cmd = "npm test -- --filter auth"
```

### How to bridge it — NOT locked (user chose "decide in planning")

The planning phase must design the bridging strategy. Two leading options:

**Option A — Smelt generates spec files + manifest:**
- Smelt creates `.assay/specs/<session-name>.toml` in the container from each `SessionDef`
- Session `spec` field → spec `description`; `harness` → `[[criteria]]` cmd; `name` → spec name
- Smelt then generates a `RunManifest` referencing those spec names
- Requires `assay init` equivalent or writing files to the correct container path
- Smelt fully owns the translation; host repo doesn't need `.assay/specs/` pre-configured

**Option B — Smelt manifest references existing specs:**
- Change Smelt's manifest format: `[[session]]` `spec` becomes an Assay spec name reference
- Host repo must already have `.assay/` with matching spec files
- Smelt just generates the `RunManifest` mapping sessions → specs
- Much simpler AssayInvoker; tighter coupling to Assay's project model

The planning phase should research how `assay init` works, what it creates, and which option is more robust before locking a choice.

### Key Assay facts for planning

- CLI: `assay run <manifest.toml>` — correct (Smelt already uses this)
- Manifest array key: `[[sessions]]` not `[[session]]` — Smelt's current generated TOML is wrong here
- `ManifestSession.spec` is a spec name string, not a description
- Assay requires `assay init` to create `.assay/` directory — container must have this
- Assay uses worktrees for agent sessions (`assay worktree create`) — bind-mount at `/workspace` must remain the project root for worktrees to work correctly
- Assay has `--base-branch`, `--timeout`, `--failure-policy`, `--merge-strategy`, `--conflict-resolution` flags — Smelt should pass relevant ones from manifest config
- Result collection: Assay manages its own merge/branch creation — Smelt's `ResultCollector` may need to adapt (Assay's result is on the base branch after merge, not a separate branch to extract)
- `assay run` returns exit 0 (all sessions merged), 1 (errors), 2 (gate failures/merge conflicts)

- D002 is firm: no Smelt crate dependency on `assay-types` — use Smelt-side serde structs that mirror Assay's format (re-verify against real schema, not import)
- D013 is firm: bind-mount at `/workspace` — must remain project root for Assay worktrees to work
- Assay binary must be installable in the target container image; test image strategy TBD in planning
- `apk add --no-cache git` in tests is fragile (requires network access to Alpine CDN) — planning should consider whether M002 introduces a pre-built test image

- Assay merge output — `assay run` exits 0/1/2; Smelt's `ResultCollector` phase may need to be adapted since Assay handles merging internally
- `assay-types` schema (source of truth at `/Users/wollax/Git/personal/assay/crates/assay-types/src/manifest.rs`) — plan must verify generated TOML against this
