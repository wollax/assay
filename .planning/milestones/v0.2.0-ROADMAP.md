# Milestone v0.2.0: Dual-Track Gates & Hardening

**Status:** ✅ SHIPPED 2026-03-08
**Goal:** Ship agent-evaluated gates (via MCP `gate_report` tool), run history persistence, required/advisory gate enforcement, session diagnostics, team context protection, and comprehensive hardening of the v0.1 foundation.
**Phases:** 11-25
**Total Plans:** 38

## Overview

v0.2.0 expanded Assay from a single-track (command-only) gate system to a dual-track quality gate platform combining deterministic shell commands with agent-evaluated criteria. It also added a complete context management subsystem inspired by Cozempic — session diagnostics, team checkpointing, composable pruning, and a guard daemon with reactive overflow recovery.

## Milestone Summary

**Key Decisions:**
- Agent gates receive evaluations via MCP, not call LLMs directly
- Self-evaluation + audit trail for v0.2; independent evaluator deferred to v0.3
- Two-tier enforcement only (required/advisory) — SonarQube validates no warnings tier
- Agent-reported gates default to advisory enforcement (trust asymmetry)
- Cozempic-inspired features in Rust (not Python) for native performance
- Guard daemon uses kqueue/inotify for sub-second reactive recovery
- Composable pruning strategies, dry-run default, team messages always protected
- Clean break for type relocation: no re-exports from assay-core

**Issues Resolved:**
- 19 test-related issues triaged and closed during Phase 19
- MCP history persistence asymmetry fixed in Phase 24
- Missing VERIFICATION.md documents backfilled in Phase 25

**Issues Deferred:**
- CLI --restore flag wiring (backup restore exists but not CLI-accessible)
- sessions_from_history() dead code (pub fn never called from CLI or MCP)
- Empty review and workflow placeholder modules in assay-core

**Technical Debt Incurred:**
- CLI --restore only lists backups; restore_backup() exists but not wired to CLI
- sessions_from_history() marked #[allow(dead_code)]
- Empty review and workflow placeholder modules (pre-v0.2.0 scaffolding)
