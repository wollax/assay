# Requirements Archive: v0.2.0 Dual-Track Gates & Hardening

**Archived:** 2026-03-08
**Status:** ✅ SHIPPED

This is the archived requirements specification for v0.2.0.

---

## Run History

- [x] **HIST-01**: Gate run results are persisted to `.assay/results/<spec>/<timestamp>.json` after every evaluation
- [x] **HIST-02**: User can view recent gate run history for a spec via `assay history <spec>`
- [x] **HIST-03**: Run history has a configurable retention policy (per-spec file count limit) enforced on save
- [x] **HIST-04**: Run history files use atomic write (tempfile-then-rename) to prevent corruption from concurrent writes

## Gate Enforcement

- [x] **ENFC-01**: Criterion has an `enforcement` field with values `required` (default) and `advisory`
- [x] **ENFC-02**: Gate evaluation summary separates required failures from advisory failures
- [x] **ENFC-03**: CLI exit code reflects only required criterion failures; advisory failures are warnings
- [x] **ENFC-04**: MCP `gate_run` response distinguishes required vs advisory results

## Agent Gate Recording

- [x] **AGNT-01**: MCP `gate_report` tool accepts agent-submitted pass/fail evaluations with structured reasoning
- [x] **AGNT-02**: `GateKind::AgentReport` variant exists for criteria evaluated by agents (not shell commands)
- [x] **AGNT-03**: Agent evaluations include `evaluator_role` metadata (`self`, `independent`, `human`)
- [x] **AGNT-04**: Agent evaluation results are persisted to run history (same store as command gate results)
- [x] **AGNT-05**: MCP `gate_history` tool allows agents to query past gate run results for a spec

## Foundation — Type System

- [x] **TYPE-01**: `GateRunSummary` and `CriterionResult` relocated from assay-core to assay-types with `Deserialize` + `JsonSchema`
- [x] **TYPE-02**: All domain types use `#[serde(skip_serializing_if)]` on optional fields
- [x] **TYPE-03**: New fields use `#[serde(default)]` for backward compatibility with existing spec/config files
- [x] **TYPE-04**: `FileExists` gate kind is wired into `evaluate()` dispatch (connect dead code)

## Foundation — Testing

- [x] **TEST-01**: MCP tool handlers have direct unit/integration tests (currently zero coverage)
- [x] **TEST-02**: Test coverage gaps from Phase 3 and Phase 6 PR reviews are addressed
- [x] **TEST-03**: All new features (history, enforcement, gate_report) have comprehensive tests

## Foundation — MCP Hardening

- [x] **MCP-01**: `gate_run` tool accepts a `timeout` parameter for agent-controlled timeouts
- [x] **MCP-02**: `resolve_working_dir` validates that the path exists before evaluation
- [x] **MCP-03**: `spec_list` handles scan errors gracefully instead of silently discarding them
- [x] **MCP-04**: Tool descriptions are accurate and field-level documentation exists on response structs

## Foundation — CLI Hardening

- [x] **CLI-01**: `main()` returns `Result` for proper error propagation
- [x] **CLI-02**: Bare `assay` invocation exits with non-zero code
- [x] **CLI-03**: `.assay` directory path is extracted to a named constant
- [x] **CLI-04**: Gate command help duplication is resolved

## Foundation — Tooling

- [x] **TOOL-01**: cargo-deny `multiple-versions` policy tightened from warn to deny
- [x] **TOOL-02**: cargo-deny `source-controls` policy tightened from warn to deny
- [x] **TOOL-03**: Dogfooding spec exists — Assay uses its own gates to enforce quality on itself

## Session Diagnostics

- [x] **SDIAG-01**: JSONL parser reads Claude Code session files from `~/.claude/projects/*/sessions/`
- [x] **SDIAG-02**: Extract exact token counts from `usage` fields in assistant messages
- [x] **SDIAG-03**: Calculate context window utilization % for model's max context (200K tokens for Opus)
- [x] **SDIAG-04**: Categorize bloat sources (progress ticks, thinking blocks, stale reads, tool output overflow, metadata, system reminders)
- [x] **SDIAG-05**: CLI `assay context diagnose` shows token usage, bloat breakdown, context %
- [x] **SDIAG-06**: CLI `assay context list` shows sessions with sizes and token counts
- [x] **SDIAG-07**: MCP `context_diagnose` tool exposes full diagnostics to agents
- [x] **SDIAG-08**: MCP `estimate_tokens` tool for quick token count + context %

## Agent Team Context Protection

- [x] **TPROT-01**: Team state extractor reads JSONL session + `~/.claude/teams/*/config.json`
- [x] **TPROT-02**: Checkpoint persists team state (agents, tasks, coordination messages) to markdown file
- [x] **TPROT-03**: CLI `assay checkpoint` command for on-demand state snapshots
- [x] **TPROT-04**: Plugin hooks trigger checkpoints on PostToolUse[Task|TaskCreate|TaskUpdate], PreCompact, Stop
- [x] **TPROT-05**: Composable pruning strategies (progress-collapse, metadata-strip, thinking-blocks, tool-output-trim, stale-reads, system-reminder-dedup)
- [x] **TPROT-06**: Team-aware pruning preserves coordination messages (Task, TeamCreate, SendMessage, TaskCreate, TaskUpdate)
- [x] **TPROT-07**: Guard daemon polls session file at configurable interval
- [x] **TPROT-08**: Soft threshold triggers gentle pruning without session reload
- [x] **TPROT-09**: Hard threshold triggers full prune + team-protect + optional session reload
- [x] **TPROT-10**: Token-based thresholds alongside file-size thresholds
- [x] **TPROT-11**: Reactive overflow recovery with file system watcher (kqueue on macOS, inotify on Linux)
- [x] **TPROT-12**: Circuit breaker prevents infinite recovery loops (configurable max recoveries in time window)
- [x] **TPROT-13**: Escalating prescriptions on repeated recoveries (gentle → standard → aggressive)
