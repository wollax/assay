  <files>crates/assay-cli/Cargo.toml, crates/assay-cli/src/main.rs</files>
  <action>
  **Cargo.toml changes:**

  Add `[[bin]]` section so the installed binary is named `assay`:
  ```toml
  [[bin]]
  name = "assay"
  path = "src/main.rs"
  ```

  **main.rs changes — Help enrichment:**

  1. Add `long_about = None` and `after_long_help` with usage examples to the top-level `Cli` struct. Examples should cover both human and agent/JSON usage:
     - `assay init`, `assay init --name my-project`
     - `assay spec list`, `assay spec show <name>`, `assay spec show <name> --json`
     - `assay gate run <spec>`, `assay gate run <spec> --json`, `assay gate run <spec> --verbose`
     - `assay mcp serve`

  2. Add `after_long_help` examples to each subcommand variant in the `Command` enum and to each leaf subcommand in `SpecCommand`, `GateCommand`, `McpCommand`. Keep examples concise and specific to that command.

  3. Doc comments on enum variants already serve as `about` text — verify they are clear. Update if needed for consistency.

  **main.rs changes — 5 CLI issue fixes (from cli-spec-cleanup):**

  1. **ANSI constant:** Extract the magic number `9` to a named constant:
     ```rust
     /// Extra bytes added by a single ANSI color sequence pair (\x1b[XXm ... \x1b[0m).
     /// \x1b[32m = 5 bytes, \x1b[0m = 4 bytes, total = 9.
     const ANSI_COLOR_OVERHEAD: usize = 9;
     ```
     Replace the hardcoded `9` in `handle_spec_show` with `ANSI_COLOR_OVERHEAD`.

  2. **NO_COLOR fix:** Change `colors_enabled()` from:
     ```rust
     std::env::var("NO_COLOR").is_err()
     ```
     to:
     ```rust
     std::env::var_os("NO_COLOR").is_none()
     ```
     This correctly handles non-UTF-8 env var values per no-color.org spec.

  3. **MCP error Debug→Display:** In the `McpCommand::Serve` arm, change `eprintln!("Error: {e:?}")` to `eprintln!("Error: {e}")` so users see human-readable error messages instead of Rust Debug output.

  4. **Init code duplication:** Replace the inline `current_dir()` call in the `Command::Init` arm with a call to `project_root()` (the existing helper).

  5. **Spec list alignment:** In `handle_spec_list()`, always pad the slug to `name_width` regardless of whether the spec has a description. Change:
     ```rust
     if spec.description.is_empty() {
         println!("  {slug}");
     } else {
         println!("  {:<width$}  {}", slug, spec.description, width = name_width);
     }
     ```
     to:
     ```rust
     if spec.description.is_empty() {
         println!("  {:<width$}", slug, width = name_width);
     } else {
         println!("  {:<width$}  {}", slug, spec.description, width = name_width);
     }
     ```
     (Both branches use the same width padding.)
  </action>

<task type="auto">
  <name>Task 2: Implement bare invocation behavior (status display and hint+help)</name>
  <files>crates/assay-cli/src/main.rs</files>
  <action>
  Replace the `None` arm in `main()` from:
  ```rust
  None => {
      println!("assay {}", env!("CARGO_PKG_VERSION"));
  }
  ```

  With logic that:
  1. Gets the current directory via `project_root()`.
  2. Checks if `.assay/` directory exists in the current directory.
  3. **If inside a project** (`.assay/` exists): call a new `show_status(root: &Path)` function that:
     - Loads config via `assay_core::config::load(&root)`
     - Prints `assay {version} -- {project_name}` header
     - Resolves specs_dir and calls `assay_core::spec::scan(&specs_dir)`
     - Lists each spec with its criteria count (total) and executable count
     - Handles errors gracefully (config load failure prints error + exits 1, spec scan failure prints warning + continues)
     - Uses `colors_enabled()` for any colored output
     - No gate execution (performance: just filesystem reads)
  4. **If outside a project** (no `.assay/`): print a hint line to stderr ("Not an Assay project. Run `assay init` to get started."), then print help via `Cli::command().print_help()`, then exit 0.

  The `show_status` function should be a standalone function following the existing pattern of `handle_spec_show`, `handle_spec_list`, etc.

  Layout for status display (Claude's discretion, but suggested approach):
  ```
  assay 0.1.0 -- my-project

  Specs:
    example-spec  (2/3 executable)

  Run `assay gate run <spec>` to evaluate gates.
  ```

  If no specs found:
  ```
  assay 0.1.0 -- my-project

  No specs found. Create one in specs/

  Run `assay init --help` for project setup details.
  ```
  </action>
  <verify>

  2. In a directory without `.assay/`: `cargo run -p assay-cli` shows hint message then help text, exits 0.
  3. Run `just ready` to confirm all checks pass.
  </verify>

  </done>
