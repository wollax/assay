<objective>
Implement a minimal MCP spike server in assay-mcp with a single hardcoded tool, wire it into the CLI as `assay mcp serve`, and validate the full JSON-RPC protocol roundtrip.

Purpose: This is a GO/NO-GO gate for the entire v0.1 architecture. It proves that rmcp 0.17 + stdio transport + Claude Code's MCP client can exchange messages successfully. If this fails, the architecture must pivot before investing in Phases 3-10.

Output: A working `assay mcp serve` command that starts an MCP server on stdio, exposes a `spike_echo` tool, and can be called by Claude Code. Spike result documented as GO or NO-GO.
</objective>

  <action>
  **Step 1: Add tracing workspace dependencies**

  In root `Cargo.toml` `[workspace.dependencies]`, add:
  ```toml
  tracing = "0.1"
  tracing-subscriber = { version = "0.3", features = ["fmt", "env-filter"] }
  ```

  **Step 2: Update assay-mcp crate dependencies**

  In `crates/assay-mcp/Cargo.toml`, add to `[dependencies]`:
  ```toml
  tracing.workspace = true
  tracing-subscriber.workspace = true
  ```

  **Step 3: Create `crates/assay-mcp/src/logging.rs`**

  Implement `pub(crate) fn init()` that initializes tracing-subscriber:
  - Output to stderr via `.with_writer(std::io::stderr)` — stdout is reserved for JSON-RPC (MCP-06)
  - Default level `warn` via EnvFilter with `RUST_LOG` override support
  - ANSI disabled (`.with_ansi(false)`) — stderr may be piped or logged
  - Use `try_init()` not `init()` to avoid panic on double-initialization

  **Step 4: Create `crates/assay-mcp/src/spike.rs`**

  Implement `SpikeServer` following the rmcp canonical pattern from RESEARCH.md:
  - `#[derive(Clone)]` struct with `tool_router: ToolRouter<Self>` field
  - `#[tool_router]` impl block with `new()` constructor calling `Self::tool_router()`
  - `spike_echo` async method annotated with `#[tool(description = "Spike: returns a hardcoded greeting to validate MCP protocol")]`
  - Returns `CallToolResult::success(vec![Content::text("spike: hello from assay")])`
  - `#[tool_handler]` on `ServerHandler` impl with `get_info()` returning:
    - `ProtocolVersion::LATEST`
    - `ServerCapabilities::builder().enable_tools().build()`
    - `Implementation::from_build_env()`
    - Instructions string identifying this as the spike server

  **Step 5: Rewrite `crates/assay-mcp/src/lib.rs`**

  Replace the doc-comment-only content with:
  - Module declarations: `mod logging;` and `mod spike;`
  - Re-export: `pub use spike::SpikeServer;`
  - `pub async fn serve() -> Result<(), Box<dyn std::error::Error>>` that:
    1. Calls `logging::init()`
    2. Logs `tracing::info!("Starting assay MCP server")`
    3. Creates `SpikeServer::new().serve(rmcp::transport::stdio()).await?`
    4. Calls `service.waiting().await?`
  - Include `inspect_err` on the serve call to log errors before propagating

  **Step 6: Update assay-cli dependencies**

  In `crates/assay-cli/Cargo.toml`, add to `[dependencies]`:
  ```toml
  assay-mcp.workspace = true
  tokio.workspace = true
  ```

  **Step 7: Wire CLI subcommand**

  Rewrite `crates/assay-cli/src/main.rs`:
  - Keep `#[derive(Parser)]` on `Cli` but add `#[command(subcommand)] command: Option<Command>`
  - Create `#[derive(Subcommand)] enum Command` with `Mcp { #[command(subcommand)] command: McpCommand }`
  - Create `#[derive(Subcommand)] enum McpCommand` with `Serve` variant
  - Change `fn main()` to `#[tokio::main] async fn main()`
  - Match on `cli.command`: `Some(Command::Mcp { command: McpCommand::Serve })` calls `assay_mcp::serve().await`
  - On error, print to stderr (`eprintln!`) and `std::process::exit(1)`
  - `None` case prints version (existing behavior)
  - Do NOT initialize tracing in main — it happens inside `assay_mcp::serve()`

  **Step 8: Verify build**

  Run `just ready`. All checks must pass (fmt-check, lint, test, deny).

  **Critical constraints:**
  - NEVER use `println!` anywhere in the MCP server path — stdout is JSON-RPC only
  - Do NOT modify any files outside the listed files (no changes to assay-types, assay-core, assay-tui)
  - Do NOT add any dependencies beyond tracing and tracing-subscriber to the workspace
  - The spike tool must have ZERO user input — hardcoded response only, no security surface
  </action>
  <verify>
  Run these commands and confirm all succeed:
  1. `just ready` — fmt-check, lint, test, deny all pass
  2. `cargo run -p assay-cli -- mcp serve < /dev/null 2>/dev/null | head -c 1` — should produce no output (server exits cleanly when stdin closes, or produces `{` if it sends a JSON-RPC message)
  3. `cargo run -p assay-cli -- --help` — shows `mcp` subcommand in help text
  4. `cargo run -p assay-cli -- mcp --help` — shows `serve` subcommand
  5. Verify no `println!` exists in assay-mcp source files
  </verify>

  <name>Task 2: Validate MCP protocol roundtrip with Claude Code and document GO/NO-GO</name>
  <files>

  <action>
  **Step 1: JSON-RPC protocol validation**

  Send a raw initialize request to verify the server speaks valid JSON-RPC:
  ```bash
  echo '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2025-03-26","capabilities":{},"clientInfo":{"name":"test","version":"0.1.0"}}}' \
    | cargo run -p assay-cli -- mcp serve 2>/dev/null \
    | head -1
  ```
  The response must be valid JSON with a `result.serverInfo` field.

  **Step 2: Stdout cleanliness verification**

  Run the server with stderr suppressed and verify ONLY JSON-RPC appears on stdout:
  ```bash
  echo '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2025-03-26","capabilities":{},"clientInfo":{"name":"test","version":"0.1.0"}}}' \
    | cargo run --release -p assay-cli -- mcp serve 2>/dev/null \
    | python3 -c "import sys,json; [json.loads(line) for line in sys.stdin if line.strip()]; print('CLEAN')"
  ```
  Must print `CLEAN` — every line on stdout is valid JSON.

  **Step 3: Claude Code integration test**

  Create a project-scoped `.mcp.json` in the repo root (do NOT commit this file):
  ```json
  {
    "mcpServers": {
      "assay": {
        "type": "stdio",
        "command": "cargo",
        "args": ["run", "--release", "-p", "assay-cli", "--", "mcp", "serve"]
      }
    }
  }
  ```

  Build the release binary first (`cargo build --release -p assay-cli`) for faster MCP startup.

  Then verify with Claude Code:
  1. Restart Claude Code or reload MCP servers
  2. Check that the `assay` MCP server appears as connected
  3. Verify `spike_echo` tool is discoverable
  4. Call the spike_echo tool and confirm it returns "spike: hello from assay"

  **Step 4: Document GO/NO-GO decision**

  Based on the three criteria:
  - Protocol roundtrip works (Step 1 succeeds)
  - No non-JSON-RPC bytes leak to stdout (Step 2 shows CLEAN)
  - Claude Code discovers and calls the tool end-to-end (Step 3 succeeds)

  If all three pass: **GO** — proceed with v0.1 architecture.
  If any has a fundamental blocker without workaround: **NO-GO** — document findings and pivot.

  Update STATE.md with the spike result (GO/NO-GO) and any observations.

  **Note:** The `.mcp.json` file should be added to `.gitignore` or left untracked — it's a local development convenience, not a committed artifact. The real plugin config is built in Phase 10.
