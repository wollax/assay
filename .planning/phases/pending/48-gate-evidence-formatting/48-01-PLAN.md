**Public API:**
```rust
/// GitHub PR body byte limit.
pub const GITHUB_BODY_LIMIT: usize = 65_536;

/// Format gate run results as markdown suitable for PR bodies.
pub fn format_gate_evidence(
    record: &GateRunRecord,
    report_path: &Path,
    char_limit: usize,
) -> FormattedEvidence
```

**Markdown structure (build with `std::fmt::Write` / `write!` / `writeln!` into a `String`):**

1. **Header:** `## Gate Results: {spec_name}`
2. **Summary stats:** `**Result:** N/M passed | F failed | S skipped` + `**Duration:** Xs` + `**Enforcement:** ...`
3. **Status table:** markdown table with columns: Status (emoji), Criterion, Enforcement, Duration. Failed rows are bold. Use `:white_check_mark:` for pass, `:x:` for fail, `:fast_forward:` for skip.
4. **Detail sections:** `<details>` blocks for:
   - All failed criteria (open by default — use `<details open>`)
   - Agent-evaluated passes (collapsed — use `<details>`)
   - Deterministic passes get NO detail section (table-only)
   - For deterministic (Command) failures: show command, exit code, stdout (if non-empty), stderr (if non-empty)
   - For deterministic (FileExists) failures: show path that was missing
   - For agent results: show evidence, reasoning, confidence
   - CRITICAL: blank line after `<summary>` tag and before `</details>` for GitHub rendering
5. **Footer:** `---` + `*Run: {run_id} | {timestamp} | assay {assay_version}*`

**Duration formatting helper:**
```rust
fn format_duration(ms: u64) -> String {
    if ms < 1_000 { format!("{ms}ms") }
    else { format!("{:.1}s", ms as f64 / 1_000.0) }
}
```

**Semantic truncation strategy:**

Build the full report first (this becomes `full_report`). Then apply truncation for `pr_body`:

1. Render the full markdown (summary + all details + footer)
2. If `.len() <= char_limit`, use as-is, `truncated: false`
3. If over limit, remove detail sections in priority order (lowest value first):
   a. Remove passing deterministic detail sections (there shouldn't be any per the rules above, but defensive)
   b. Remove passing agent detail sections (collapsed ones), last-to-first
   c. Remove failing detail sections, last-to-first
4. If still over limit with NO detail sections, truncate by removing table rows from the bottom until it fits
5. Append truncation notice: `\n\n> **Note:** This report was truncated to fit GitHub's character limit. Full report: \`{report_path}\``

Implementation approach: Build the summary section and footer as one string. Build each detail section as a separate string in a `Vec`. Classify each detail section by priority (agent-pass, failure). Assemble full report. For truncation, try progressively removing sections by priority until under limit. Use `.len()` (byte length) for all size checks.

Handle edge cases:
- Empty results vec → produce a valid table with no rows and no detail sections
- Skipped criteria (result: None) → table row with skip emoji, no detail section
- Long reasoning/evidence from agents that exceeds limit on its own → the truncation strategy will remove the entire detail section, which handles this
