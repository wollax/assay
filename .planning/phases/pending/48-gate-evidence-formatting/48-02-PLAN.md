<task type="auto">
  <name>Task 1: Formatting and detail section tests</name>
  <files>crates/assay-core/src/gate/evidence.rs</files>
  <action>
Add `#[cfg(test)] mod tests` to `evidence.rs` with test helper functions and formatting tests.

**Test helpers:**
- `make_test_record(spec_name, results) -> GateRunRecord` — builds a record with known timestamp, run_id, and assay_version for deterministic output
- `make_command_pass(name, cmd, stdout) -> CriterionResult` — deterministic pass
- `make_command_fail(name, cmd, exit_code, stderr) -> CriterionResult` — deterministic fail
- `make_agent_pass(name, evidence, reasoning) -> CriterionResult` — agent pass
- `make_agent_fail(name, evidence, reasoning) -> CriterionResult` — agent fail
- `make_skipped(name) -> CriterionResult` — skipped criterion (result: None)
- `make_file_exists_pass(name, path) -> CriterionResult` — FileExists pass
- `make_file_exists_fail(name, path) -> CriterionResult` — FileExists fail

**Formatting tests:**

1. `test_format_empty_results` — zero criteria produces valid markdown with header, empty table, and footer. No detail sections. `truncated: false`.

2. `test_format_all_pass_deterministic` — 3 passing Command criteria. Table has 3 rows with `:white_check_mark:`. No detail sections (deterministic passes are table-only). `truncated: false`.

3. `test_format_mixed_results` — mix of: command pass, command fail, agent pass, agent fail, skipped. Verify:
   - Table has correct emoji per row
   - Failed rows are bold in table
   - Failed command has `<details open>` with exit code, stderr
   - Failed agent has `<details open>` with evidence, reasoning
   - Agent pass has `<details>` (collapsed) with evidence, reasoning, confidence
   - Deterministic pass has NO detail section
   - Skipped has skip emoji, no detail section

4. `test_format_detail_sections_have_blank_lines` — verify that every `<details>` block has a blank line after `<summary>` and before `</details>` (P2 pitfall). Parse the output and check structure.

5. `test_format_file_exists_fail` — FileExists failure detail section shows the missing path.

6. `test_format_includes_footer` — output ends with `---` and run metadata line containing run_id, timestamp, and assay_version.

7. `test_format_enforcement_summary` — verify enforcement line shows required/advisory breakdown from the `EnforcementSummary`.

8. `test_format_duration_helper` — unit test for `format_duration()`: <1000ms shows "Xms", >=1000ms shows "X.Ys".

<task type="auto">
  <name>Task 2: Truncation and persistence tests</name>
  <files>crates/assay-core/src/gate/evidence.rs</files>
  <action>
Add truncation and persistence tests to the existing `#[cfg(test)] mod tests` block.

**Truncation tests:**

1. `test_no_truncation_within_limit` — full output fits within limit. `truncated: false`. `pr_body == full_report`.

2. `test_truncation_removes_agent_passes_first` — create a record with 1 failure + 3 agent passes. Set char_limit just below the full output size. Verify:
   - `truncated: true`
   - `pr_body` still contains the failure detail section
   - `pr_body` is missing at least one agent pass detail section
   - `pr_body` contains the truncation notice with report path
   - `pr_body.len() <= char_limit`

3. `test_truncation_removes_failures_last` — create a record with 3 failures + 1 agent pass. Set char_limit tight. Verify failures are removed only after agent passes are gone.

4. `test_truncation_preserves_summary_table` — even with aggressive truncation, the summary table and header are always present. Set limit very tight (but big enough for summary).

5. `test_truncation_notice_includes_report_path` — when truncated, the output contains the report_path string.

6. `test_full_report_is_never_truncated` — regardless of char_limit, `full_report` always contains all detail sections and is identical to what would be produced with an unlimited limit.

7. `test_truncation_uses_byte_length` — create output with multi-byte UTF-8 characters in criterion names. Verify truncation checks `.len()` (bytes) not `.chars().count()`.

**Persistence tests:**

8. `test_save_report_creates_file` — save to a tempdir, verify file exists at expected path `.assay/reports/<spec>//<run-id>.md`.

9. `test_save_report_creates_directories` — reports dir doesn't exist yet, save creates it.

10. `test_save_report_rejects_path_traversal` — spec name with `..` or `/` is rejected.

11. `test_save_report_content_matches` — saved file content equals `evidence.full_report`.
