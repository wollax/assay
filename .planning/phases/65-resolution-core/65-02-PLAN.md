  <behavior>
    **Happy paths:**
    - resolve() with no extends/include returns ResolvedGate with own criteria marked CriterionSource::Own
    - resolve() with extends loads parent, includes parent criteria as CriterionSource::Parent
    - resolve() with include loads libraries, includes library criteria as CriterionSource::Library
    - resolve() with extends + include merges all three sources in order: parent -> libraries -> own

    **Own-wins merge semantics:**
    - Parent criterion "compiles" + own criterion "compiles" -> resolved has only own "compiles" (source=Own)
    - Library criterion "tests-pass" + own criterion "tests-pass" -> resolved has only own "tests-pass" (source=Own)
    - Library A criterion "lint" + Library B criterion "lint" -> resolved has Library B "lint" (later library wins)
    - Parent criterion "build" not overridden -> resolved has parent "build" (source=Parent)

    **Merge ordering (Vec output):**
    - Criteria order in resolved Vec: parent criteria first (non-overridden), then library criteria (non-overridden), then own criteria
    - Within each group, original order preserved

    **Cycle detection:**
    - Self-extend: gate_slug == extends slug -> Err(CycleDetected)
    - Mutual cycle: gate A extends B, B extends A -> Err(CycleDetected)
    - Uses gate_slug parameter for identity (not gate.name field) per RESEARCH.md Pitfall 4

    **Single-level extends:**
    - Parent's extends is ignored (B's extends = C is NOT followed) per locked decision
    - Parent's include is ignored (parent criteria taken as-is) per locked decision

    **Slug validation:**
    - resolve() validates extends slug before loading -> Err(InvalidSlug)
    - resolve() validates each include slug before loading -> Err(InvalidSlug)

    **Error paths:**
    - Nonexistent parent gate slug -> Err(ParentGateNotFound)
    - Nonexistent library slug -> load_library closure returns error, resolve propagates it
    - Fail on first error (no multi-error collection) per locked decision

    **Edge cases:**
    - Gate with empty criteria + extends with criteria -> all parent criteria in output
    - Gate with criteria + empty include list -> no library criteria
    - Gate with no extends, no include -> pure own criteria
  </behavior>

Add `pub fn resolve()` to `crates/assay-core/src/spec/compose.rs`:

```rust
pub fn resolve(
    gate: &GatesSpec,
    gate_slug: &str,
    load_gate: impl Fn(&str) -> Result<GatesSpec>,
    load_library: impl Fn(&str) -> Result<CriteriaLibrary>,
) -> Result<ResolvedGate>
```

**Algorithm:**

1. **Validate slugs**: If `extends` is Some, validate_slug it. For each `include` entry, validate_slug it.

2. **Cycle detection**: If `extends` is Some:
   - If `extends_slug == gate_slug` -> `Err(CycleDetected { gate_slug, parent_slug: extends_slug })`
   - Load parent via `load_gate(extends_slug)?`, wrapping errors to `ParentGateNotFound` if load fails.
   - If `parent.extends == Some(gate_slug)` -> `Err(CycleDetected { gate_slug, parent_slug: extends_slug })`

3. **Build merge map**: Use `IndexMap<String, ResolvedCriterion>` (or equivalent ordered insert) to preserve insertion order while allowing key-based overwrites:
   - Insert parent criteria (if extends) with `CriterionSource::Parent { gate_slug: extends_slug }`
   - Insert library criteria (for each include slug) with `CriterionSource::Library { slug }`
   - Insert own criteria with `CriterionSource::Own`
   - On name collision, later entry wins (overwrites silently) — this gives own-wins semantics naturally.

   Note: Use a `Vec` + position tracking if avoiding `indexmap` dependency. Research shows no `indexmap` in workspace — use a `Vec<(String, ResolvedCriterion)>` with dedup by name, or `std::collections::HashMap` + ordered Vec rebuild. Simplest: build a `Vec<ResolvedCriterion>` and then dedup by name keeping last occurrence. Alternative: use a temporary `HashMap<String, usize>` tracking name->position for O(1) overwrites.

   Recommended approach: Use a `Vec<ResolvedCriterion>` as accumulator. After collecting all criteria, walk backwards to find the LAST occurrence of each name, then collect in forward order skipping earlier duplicates. This preserves source-group ordering (parent first, then libraries, then own) while ensuring own-wins.

   Actually, simpler: use a `HashMap<String, ResolvedCriterion>` for dedup, then rebuild ordered output as: parent criteria (in order, if still in map) + library criteria (in order, if still in map) + own criteria (in order). After all insertions, iterate the original order and collect only entries whose name is still mapped to their source.

   Simplest correct approach: Insert all criteria into a `Vec`, then reverse-dedup by name (keep last, which is always own if present). To do this: iterate in reverse, track seen names in a `HashSet`, keep only first-seen (which is last in original order). Then reverse the result to restore order.

4. **Build ResolvedGate**:
   ```rust
   ResolvedGate {
       gate_slug: gate_slug.to_string(),
       parent_slug: gate.extends.clone(),
       included_libraries: gate.include.clone(),
       criteria: deduped_criteria,
   }
   ```

**Test structure**: All tests use closure pairs that return in-memory GatesSpec/CriteriaLibrary values (no filesystem). This makes tests fast and deterministic.
