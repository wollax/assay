<task type="auto">
  <name>Task 1: Triage all open issues</name>
  <files>.planning/issues/open/*, .planning/issues/closed/*</files>
  <action>
  Process all issues in `.planning/issues/open/`. Follow this procedure for EACH issue:

  **Reading:**
  1. Read the issue file (YAML frontmatter + Problem/Solution sections)
  2. Note the `area`, `severity`, `created` date, and referenced files/lines

  **Verification (CONTEXT.md decision #9 — non-negotiable):**
  3. Read the ACTUAL current source code referenced by the issue
  4. Determine if the problem described still exists in current code
  5. Do NOT just match against phase timelines — verify the code

  **Resolution decision (CONTEXT.md decision #7):**
  - If the code no longer exhibits the issue: CLOSE it
    - Create `.planning/issues/closed/` directory if it doesn't exist
    - Move the file: `mv .planning/issues/open/{filename} .planning/issues/closed/{filename}`
    - No need to modify the file contents — the move itself is the closure action

  **Categorization (for issues that remain open):**
  - Assign priority tier (CONTEXT.md decision #11):
    - **must-fix**: Correctness issues, API design problems, missing validation
    - **should-fix**: Code quality, duplication, naming, ergonomics
    - **nice-to-have**: Doc comments, minor tests, cosmetic improvements
  - Assign target milestone (CONTEXT.md decision #12):
    - **v0.2.1**: must-fix items
    - **v0.3.0**: should-fix items
    - **backlog**: nice-to-have items
  - Note functional area from the issue's `area` field

  **Processing order (RESEARCH.md recommendation):**
  - Start with 2026-03-01 and 2026-03-02 issues (40 total, ~45-70% expected closure rate)
  - Then 2026-03-04 issues (40 total)
  - Then 2026-03-05 and later issues (48 total)
  - Within each date group, batch by functional area to minimize file-switching

  **Known resolutions to check first (from RESEARCH.md):**
  - `deny-multiple-versions` — resolved by Phase 19-01
  - `deny-source-controls` — resolved by Phase 19-01
  - `dogfood-checkpoint` — resolved by Phase 19-03
  - `mcp-tool-handler-test-coverage` — resolved by Phase 19-02
  - `test-coverage-gaps-phase3` — resolved by Phase 19-02
  - `test-coverage-gaps-phase6` — partially resolved by Phase 19-02

  **Pitfalls to avoid (from RESEARCH.md):**
  - False closure: check the underlying problem, not just the specific symptom
  - Duplicate issues: note overlapping issues (e.g., `history-io-error-conflation` and `history-serde-json-error-conflation`)
  - Phase 19 already closed 19 issues — the 128 remaining are post-that-triage
  - Stale file:line references — use them as hints, not addresses
  - Severity field inconsistency — some use `severity:`, others `priority:`

  **Important:** Keep all valid observations open regardless of how minor (CONTEXT.md decision #8). Only close issues where the code genuinely no longer exhibits the problem.
  </action>
  <verify>
  - `.planning/issues/closed/` directory exists and contains moved issue files
  - No issue file exists in both open/ and closed/
  - Remaining open issues have been read and mentally categorized
  </verify>
  <done>
  All 128 issues have been read and verified against current source. Resolved issues moved to closed/. Remaining issues categorized by priority and milestone.
  </done>
</task>
