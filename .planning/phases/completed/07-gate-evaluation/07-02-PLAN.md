  <action>
Add the `assay gate run <spec>` CLI command with full streaming output and all flags.

**Clap subcommand structure:**

1. Add `Gate` variant to the `Command` enum:
   ```rust
   Gate {
       #[command(subcommand)]
       command: GateCommand,
   },
   ```

2. Add `GateCommand` enum:
   ```rust
   #[derive(Subcommand)]
   enum GateCommand {
       /// Run all executable criteria for a spec
       Run {
           /// Spec name (filename without .toml extension)
           name: String,
           /// Override timeout for all criteria (seconds)
           #[arg(long)]
           timeout: Option<u64>,
           /// Show evidence for all criteria, not just failures
           #[arg(short, long)]
           verbose: bool,
           /// Output as JSON instead of streaming display
           #[arg(long)]
           json: bool,
       },
   }
   ```

3. Wire the match arm in `main()`:
   ```rust
   Some(Command::Gate { command }) => match command {
       GateCommand::Run { name, timeout, verbose, json } => {
           handle_gate_run(&name, timeout, verbose, json);
       }
   },
   ```

**`handle_gate_run` function implementation:**

4. Load config and resolve specs_dir (same pattern as `handle_spec_show`).

5. Load the spec file. On NotFound, use the same error pattern as spec show: `Error: spec '{name}' not found in {specs_dir}` with exit 1.

6. Resolve the working directory for gate evaluation:
   - If config has `[gates].working_dir`, use that (resolve relative to project root).
   - Otherwise, use the project root itself.
   - The working_dir is ALWAYS explicit — never inherit process CWD. Using project_root() as the default satisfies GATE-04 because it's an explicit choice, not an implicit inheritance.

7. Extract `config_timeout` from `config.gates.as_ref().map(|g| g.default_timeout)`.

**JSON output path (`--json`):**

8. If `--json` is set, do NOT stream progress. Instead:
   - Call `assay_core::gate::evaluate_all()` to get the GateRunSummary.
   - Serialize the entire GateRunSummary to JSON with `serde_json::to_string_pretty()`.
   - Print to stdout with `println!`.
   - Exit with code 1 if `summary.failed > 0`, else 0.

**Streaming display path (default):**

9. Do NOT call `evaluate_all()` for the streaming path — instead, iterate criteria manually to show progress per-criterion. This allows showing the "running" state before each evaluation completes.

10. For each criterion in spec.criteria:
    - If criterion has no `cmd`: record as skipped, continue (don't print during execution).
    - If criterion has `cmd`:
      a. Print running state: `eprint!("\r\x1b[K  {} ... running", criterion.name)` (stderr, overwritable).
      b. Call `assay_core::gate::evaluate(&criterion, &working_dir, timeout)` where timeout is `resolve_timeout(cli_timeout, criterion.timeout, config_timeout)`.
      c. On success, replace the line with pass/fail:
         - Pass: `eprintln!("\r\x1b[K  {} ... {}", name, green("ok"))`
         - Fail: `eprintln!("\r\x1b[K  {} ... {}", name, red("FAILED"))`
      d. On Err (spawn failure), show as FAILED with error message.
      e. If failed (or verbose), print evidence indented under the criterion:
         - stdout (if non-empty): `eprintln!("    stdout: {}", result.stdout.trim())`
         - stderr (if non-empty): `eprintln!("    stderr: {}", result.stderr.trim())`
         - For multi-line evidence, indent each line with 4 spaces.
      f. Collect into results Vec for the summary.

11. After all criteria, print a blank line then the summary:
    ```
    Results: N passed, M failed, K skipped (of T total)
    ```
    Color the counts: passed in green, failed in red (if > 0), skipped in yellow (if > 0).
    Respect `NO_COLOR` via the existing `colors_enabled()` function.

12. Exit code: `std::process::exit(1)` if any criteria failed, `std::process::exit(0)` otherwise.

**Color helpers:**

13. Add helper functions (or extend existing) for red/green/yellow coloring:
    - `format_pass(color: bool) -> &'static str` — green "ok" or plain "ok"
    - `format_fail(color: bool) -> &'static str` — red "FAILED" or plain "FAILED"
    These follow the existing `format_criteria_type` pattern but are simpler.

**Edge cases to handle:**

- Spec with zero executable criteria: print "No executable criteria found" and exit 0.
- All criteria pass: exit 0.
- Mix of pass/fail: always run all (never early exit), exit 1.
- Truncated output: if GateResult.truncated is true, note in evidence display (e.g., "[output truncated]").
- `--timeout 0` or negative: `resolve_timeout` already enforces 1s minimum, but the CLI should validate and warn if timeout < 1: `eprintln!("Warning: timeout must be at least 1 second, using 1s")`. Actually, since resolve_timeout handles this in core, the CLI can just pass it through.

**Important constraints:**
- Progress goes to stderr (`eprint!`/`eprintln!`), final summary goes to stderr, JSON goes to stdout. This matches the convention from 07-CONTEXT.md: "Use eprint!/eprintln! for progress (stderr), println! for final summary and --json (stdout)." Wait — re-reading: the summary is the final result, so for non-JSON mode, print the summary line to stdout with `println!` so it's capturable. The streaming progress during execution goes to stderr.
- Actually, looking at the CONTEXT more carefully: "Stream progress during execution" (stderr) and "show a summary table at the end" (stdout). So: streaming lines = stderr, final summary = stdout. JSON = stdout. This means `println!` for the summary and `eprintln!` for per-criterion progress.
- Respect NO_COLOR for all ANSI codes.
- Do NOT add new dependencies to assay-cli.
