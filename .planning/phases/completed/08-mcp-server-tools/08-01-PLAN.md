**Step 2 — Create server.rs:**

Create `crates/assay-mcp/src/server.rs` with the complete MCP server implementation. Follow the patterns from 08-RESEARCH.md precisely. The module contains:

**Imports:**
```rust
use std::path::{Path, PathBuf};

use rmcp::{
    ErrorData as McpError, ServerHandler, ServiceExt,
    handler::server::tool::ToolRouter,
    handler::server::wrapper::Parameters,
    model::*,
    tool, tool_handler, tool_router,
    transport::io::stdio,
};
use schemars::JsonSchema;
use serde::{Deserialize, Serialize};

use assay_types::{Config, Spec};
```

**Parameter structs (with JsonSchema for agent discoverability):**

1. `SpecGetParams` — single `name: String` field with `#[schemars(description = "Spec name (filename without .toml extension, e.g. 'auth-flow')")]`.

2. `GateRunParams` — `name: String` with schemars description, plus `include_evidence: bool` with `#[serde(default)]` and `#[schemars(description = "Include full stdout/stderr evidence per criterion (default: false). When false, returns summary only with pass/fail, exit code, and duration. When true, adds stdout and stderr fields to each criterion result.")]`.

3. `spec_list` takes no parameters — rmcp generates an empty object schema automatically when there's no `Parameters<T>` argument.

**Response structs (with Serialize + skip_serializing_if for bounded responses):**

4. `SpecListEntry` — `name: String`, `description: String` with `#[serde(skip_serializing_if = "String::is_empty")]`, `criteria_count: usize`.

5. `GateRunResponse` — `spec_name: String`, `passed: usize`, `failed: usize`, `skipped: usize`, `total_duration_ms: u64`, `criteria: Vec<CriterionSummary>`.

6. `CriterionSummary` — `name: String`, `status: String` (one of "passed", "failed", "skipped"), `exit_code: Option<i32>` with skip_serializing_if, `duration_ms: Option<u64>` with skip_serializing_if, `reason: Option<String>` with skip_serializing_if (stderr first line for failures only in summary mode), `stdout: Option<String>` with skip_serializing_if (only when include_evidence), `stderr: Option<String>` with skip_serializing_if (only when include_evidence).

**Server struct:**

7. `AssayServer` struct with `tool_router: ToolRouter<Self>`. Derive `Clone`. Implement `Default` delegating to `new()`.

**`#[tool_router] impl AssayServer` block with tools:**

8. `pub fn new() -> Self` that initializes `tool_router: Self::tool_router()`.

9. `spec_list` tool:
   - Description: `"List all specs in the current Assay project. Returns an array of {name, description, criteria_count} objects. Use this to discover available specs before calling spec_get or gate_run."`
   - No parameters.
   - Implementation: `resolve_cwd()` -> `load_config()` -> `assay_core::spec::scan()` -> map each `(slug, spec)` to `SpecListEntry { name: slug, description: spec.description, criteria_count: spec.criteria.len() }` -> serialize to JSON -> `CallToolResult::success(vec![Content::text(json)])`.
   - On scan errors: include scan warnings in the response (append to the JSON or note in a separate field). For v0.1, if scan itself fails (e.g., specs dir doesn't exist), return domain_error. If individual files fail to parse, include successful specs and ignore the errors (matching CLI behavior).

10. `spec_get` tool:
    - Description: `"Get a spec by name. Returns the full spec definition as JSON including name, description, and all criteria with their commands and timeouts. Use spec_list first to find available spec names."`
    - Parameters: `Parameters<SpecGetParams>`.
    - Implementation: `resolve_cwd()` -> `load_config()` -> `load_spec()` -> `serde_json::to_string(&spec)` -> `CallToolResult::success(vec![Content::text(json)])`.
    - On spec not found: return `domain_error` with the AssayError message (which includes the spec name and directory).

11. `gate_run` tool:
    - Description: `"Run quality gate checks for a spec. Evaluates all executable criteria (shell commands) and returns pass/fail status per criterion with aggregate counts. Set include_evidence=true for full stdout/stderr output per criterion."`
    - Parameters: `Parameters<GateRunParams>`.
    - Implementation:
      a. `resolve_cwd()` -> `load_config()` -> `load_spec()`.
      b. `resolve_working_dir(&cwd, &config)` for the gate execution directory.
      c. Extract `config_timeout = config.gates.as_ref().map(|g| g.default_timeout)`.
      d. Clone `spec` and `working_dir` into owned values for the `spawn_blocking` closure.
      e. Call `tokio::task::spawn_blocking(move || assay_core::gate::evaluate_all(&spec_owned, &working_dir_owned, None, config_timeout)).await`.
      f. Map `JoinError` to `McpError::internal_error("gate evaluation panicked: {e}", None)`.
      g. Format the `GateRunSummary` into `GateRunResponse` using `format_gate_response(&summary, include_evidence)`.
      h. Serialize to JSON and return as `CallToolResult::success`.

**`#[tool_handler] impl ServerHandler for AssayServer` block:**

12. `get_info()` returning `ServerInfo` with:
    - `protocol_version: ProtocolVersion::LATEST`
    - `capabilities: ServerCapabilities::builder().enable_tools().build()`
    - `server_info: Implementation::from_build_env()`
    - `instructions: Some("Assay development kit. Manages specs (what to build) and gates (quality checks). Use spec_list to discover specs, spec_get to read one, gate_run to evaluate criteria.".to_string())`

**Private helper functions:**

13. `resolve_cwd() -> Result<PathBuf, McpError>` — `std::env::current_dir()` mapped to `McpError::internal_error`.

14. `load_config(cwd: &Path) -> Result<Config, CallToolResult>` — `assay_core::config::load(cwd)` mapped through `domain_error`.

15. `load_spec(cwd: &Path, config: &Config, name: &str) -> Result<Spec, CallToolResult>` — resolve specs_dir from config, build spec path, call `assay_core::spec::load`. Map errors through `domain_error`.

16. `resolve_working_dir(cwd: &Path, config: &Config) -> PathBuf` — match on `config.gates.as_ref().and_then(|g| g.working_dir.as_deref())`, resolve relative to cwd. This matches the CLI pattern in `handle_gate_run`.

17. `domain_error(err: &assay_core::AssayError) -> CallToolResult` — `CallToolResult::error(vec![Content::text(err.to_string())])`. This ensures agents see the error and can self-correct.

18. `format_gate_response(summary: &assay_core::gate::GateRunSummary, include_evidence: bool) -> GateRunResponse` — Maps `GateRunSummary` to the bounded `GateRunResponse` struct:
    - For each `CriterionResult` in summary:
      - If `result` is None: status = "skipped", no exit_code/duration/reason/stdout/stderr.
      - If `result` is Some and `passed`: status = "passed", include exit_code and duration_ms. Include stdout/stderr only if `include_evidence`.
      - If `result` is Some and not `passed`: status = "failed", include exit_code, duration_ms, and `reason` (first non-empty line from stderr, or "unknown" if stderr is empty). Include stdout/stderr only if `include_evidence`.

**Important pattern for tool handlers and helpers with two error types:**

The tool handler functions return `Result<CallToolResult, McpError>`. The helpers `load_config` and `load_spec` return `Result<T, CallToolResult>` (not `McpError`) because domain errors should be `isError: true` tool results. In the tool handler, use this pattern:

```rust
let cwd = resolve_cwd()?; // McpError propagates via ?
let config = match load_config(&cwd) {
    Ok(c) => c,
    Err(err_result) => return Ok(err_result), // domain error -> Ok(CallToolResult { is_error: true })
};
```

This is critical — domain errors must be `Ok(CallToolResult::error(...))`, not `Err(McpError)`.

**`pub async fn serve()` function in server.rs:**

19. `pub async fn serve() -> Result<(), Box<dyn std::error::Error>>` — Create `AssayServer::new()`, call `.serve(stdio()).await?`, then `.waiting().await?`. Same pattern as the spike, different server type.
