**Step 2 — Add integration tests to server.rs:**

Add integration tests to the `#[cfg(test)] mod tests` block in `crates/assay-mcp/src/server.rs`. These tests exercise the tool handlers by constructing an `AssayServer`, creating temporary project directories, and calling the tools directly (not through MCP transport — that level of integration testing is deferred to manual verification).

The approach: since tool methods on `AssayServer` are `async fn` that read from CWD (`std::env::current_dir()`), tests must `std::env::set_current_dir()` to a tempdir with a valid `.assay/` layout. **Important:** this makes these tests non-parallelizable. Use `#[serial]` via the `serial_test` crate if it's in workspace deps, or document that these tests must run single-threaded. Alternatively, if setting CWD is too fragile, test the helper functions and response formatting directly (no CWD dependency) and leave full tool handler tests to manual verification.

**Preferred approach: Test helpers and formatting directly (no CWD manipulation), plus manual verification for full flow.** This is safer and more maintainable. The unit tests from Plan 01 cover formatting. Add these additional tests:

**Helper function tests (can use explicit paths, no CWD dependency):**

1. `test_load_config_valid_project`: Create a tempdir with `.assay/config.toml`, call `load_config()` (note: this requires CWD — so instead, test the pattern by calling `assay_core::config::load()` directly, which the helper wraps). Actually, the helpers `load_config` and `load_spec` take `cwd: &Path` explicitly — they do NOT use `std::env::current_dir()`. Only `resolve_cwd()` does. So we can test helpers directly:
   - Create tempdir with valid `.assay/config.toml` and `.assay/specs/test.toml`.
   - Call `load_config(tempdir.path())` and assert it returns Ok.
   - Call `load_spec(tempdir.path(), &config, "test")` and assert it returns Ok with correct spec data.

2. `test_load_config_missing_project`: Create empty tempdir. Call `load_config(tempdir.path())`. Assert it returns Err (which is a CallToolResult with isError). Verify the error text mentions "config" and path.

3. `test_load_spec_not_found`: Create tempdir with valid config but no spec files. Call `load_spec(tempdir.path(), &config, "nonexistent")`. Assert it returns Err with helpful message mentioning the spec name.

4. `test_resolve_working_dir_default`: Config with no `[gates]` section. Assert `resolve_working_dir(cwd, &config)` returns `cwd`.

5. `test_resolve_working_dir_relative`: Config with `gates.working_dir = Some("subdir")`. Assert returns `cwd.join("subdir")`.

6. `test_resolve_working_dir_absolute`: Config with `gates.working_dir = Some("/tmp/custom")`. Assert returns `/tmp/custom`.

**Response formatting integration test:**

7. `test_spec_list_entry_serialization`: Create `SpecListEntry` values, serialize to JSON, verify shape matches expected wire format. Verify description is omitted when empty.

8. `test_gate_run_response_serialization`: Create a `GateRunResponse` with mixed results, serialize to JSON, verify wire format. Verify skip_serializing_if works (no null fields in JSON for skipped criteria).

**Important:** These tests do NOT require `serial_test` or CWD manipulation. They test the components that the tool handlers compose. The full tool handler flow (CWD -> config -> spec -> result) is verified manually in Task 2.

Add `tempfile` to assay-mcp dev-dependencies if not already present (check `Cargo.toml`). Since `tempfile` is a workspace dependency, add it as:
```toml
[dev-dependencies]
tempfile.workspace = true
```

**Scenario 1 — Clean JSON-RPC start (MCP-05):**

Start the server and verify the first byte on stdout is `{`:

```bash
echo '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2024-11-05","capabilities":{},"clientInfo":{"name":"test","version":"0.1"}}}' | cargo run -p assay-cli -- mcp serve 2>/dev/null | head -c 1
```

Expected: `{` (no clap banner, no tracing output on stdout).

**Scenario 2 — tools/list returns three tools (MCP-08):**

Send an `initialize` + `tools/list` request sequence and verify the response lists three tools with self-documenting descriptions:

```bash
printf '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2024-11-05","capabilities":{},"clientInfo":{"name":"test","version":"0.1"}}}\n{"jsonrpc":"2.0","method":"notifications/initialized"}\n{"jsonrpc":"2.0","id":2,"method":"tools/list","params":{}}' | cargo run -p assay-cli -- mcp serve 2>/dev/null
```

Expected: Response for id=2 contains `tools` array with three entries. Each tool has `name` and `description`. Tool names are `spec_get`, `spec_list`, `gate_run`. Descriptions are substantive (not just the tool name repeated).

**Scenario 3 — spec_list returns spec entries (MCP-04):**

```bash
printf '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2024-11-05","capabilities":{},"clientInfo":{"name":"test","version":"0.1"}}}\n{"jsonrpc":"2.0","method":"notifications/initialized"}\n{"jsonrpc":"2.0","id":2,"method":"tools/call","params":{"name":"spec_list","arguments":{}}}' | cargo run -p assay-cli -- mcp serve 2>/dev/null
```

Expected: Response contains JSON array with entries for "auth-flow" and "empty". Each entry has name, criteria_count. "auth-flow" has criteria_count=3, "empty" has criteria_count=1.

**Scenario 4 — spec_get returns full spec (MCP-02):**

```bash
printf '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":"2024-11-05","capabilities":{},"clientInfo":{"name":"test","version":"0.1"}}}\n{"jsonrpc":"2.0","method":"notifications/initialized"}\n{"jsonrpc":"2.0","id":2,"method":"tools/call","params":{"name":"spec_get","arguments":{"name":"auth-flow"}}}' | cargo run -p assay-cli -- mcp serve 2>/dev/null
```

Expected: Response contains the full auth-flow spec as JSON with name, description, and all three criteria including cmd fields.

**Scenario 5 — spec_get with nonexistent spec returns tool error:**

Expected: Response contains JSON with spec_name "auth-flow", passed=2, failed=0, skipped=1, and criteria array. Criteria for "unit-tests" and "type-check" have status "passed". "review-checklist" has status "skipped". No stdout/stderr fields in default mode.
