  <name>Task 1: Wire ERR-01 — Exit code detection into gate evaluation</name>
  <files>crates/assay-core/src/gate/mod.rs, crates/assay-cli/src/commands/gate.rs</files>
  <action>
**Gate result processing (exit code 127/126):**

The common case for command-not-found is exit code 127 from `sh -c`. The GateResult already captures `exit_code: Option<i32>`. Wire the detection into two locations:

1. **`evaluate_criteria()` in gate/mod.rs** — In the `Ok(gate_result)` arm where `!gate_result.passed`, check if `gate_result.exit_code` is 127 or 126. If so, use `classify_exit_code()` + `format_command_error()` to produce an enriched stderr message. Append the actionable message to the existing stderr (don't replace — the shell's error message like "sh: cargo: not found" is still useful context). Example: append `"\ncommand 'cargo' not found. Is it installed and in PATH?"` to the GateResult stderr field.

   **Important:** The `cmd` string is needed for `extract_binary()`. In `evaluate_criteria()`, the cmd is available from `criterion.cmd.as_deref()`. Only apply when the criterion has a cmd (not file-exists or always-pass gates).

2. **`stream_criterion()` in gate.rs CLI** — In the `Ok(result)` arm where `!result.passed`, check for exit 127/126. The streaming display currently shows `format_fail(cfg.color)`. For 127/126, additionally print the actionable message below the FAIL line (similar to how evidence is shown). Use the functions from Plan 01.

   Also in the `Err(err)` arm — for `AssayError::GateExecution` with `source.kind() == NotFound` or `PermissionDenied`, use `format_command_error()` instead of `eprintln!("    error: {err}")`. This handles the rare case where `sh` itself is missing.

**Spawn error handling (rare):**

3. The `.spawn()` error path in `evaluate_command()` already creates `AssayError::GateExecution`. When this error reaches `evaluate_criteria()` (the `Err(err)` arm at line 214), the error is formatted as `"gate evaluation error: {err}"` in the stderr field. Check if the error is `GateExecution` with `NotFound`/`PermissionDenied` source kind and use the improved message.

   Simplest approach: add a method `AssayError::gate_execution_display(&self) -> String` that returns the improved message for GateExecution with NotFound/PermissionDenied, or falls back to `self.to_string()`. Use this in both the `evaluate_criteria` error formatting and the CLI `stream_criterion` error display.

<task type="auto">
  <name>Task 2: Wire ERR-02 — Spec-not-found diagnostics into all call sites</name>
  <files>crates/assay-core/src/spec/mod.rs, crates/assay-cli/src/commands/gate.rs, crates/assay-cli/src/commands/spec.rs, crates/assay-mcp/src/server.rs</files>
  <action>
**Core function:**

1. Add `pub fn load_spec_entry_with_diagnostics(slug: &str, specs_dir: &Path) -> Result<SpecEntry>` to `crates/assay-core/src/spec/mod.rs`:
   - Call `load_spec_entry(slug, specs_dir)`
   - On `Ok(entry)` -> return Ok
   - On `Err(AssayError::SpecNotFound { name, specs_dir })`:
     - Call `scan(&specs_dir)`. If scan fails, fall back to returning the original `SpecNotFound` error (bare message).
     - Extract `available: Vec<String>` from `scan_result.entries.iter().map(|e| e.slug().to_string()).collect()`
     - Extract `invalid: Vec<String>` from `scan_result.errors` — extract slug/filename from each error (pattern match on SpecParse, GatesSpecParse, etc. to get path, then extract stem)
     - Call `find_fuzzy_match(&name, &available)` for suggestion
     - Return `Err(AssayError::SpecNotFoundDiagnostic { name, specs_dir, available, invalid, suggestion })`
   - On other errors -> propagate as-is

**CLI call sites (replace bail! patterns):**

2. **gate.rs** — 4 locations that catch `SpecNotFound` and `bail!`:
   - `handle_gate_run()` line ~444: replace `load_spec_entry` with `load_spec_entry_with_diagnostics`, remove the `SpecNotFound` match arm. Let the enriched error propagate via `?` (it will be converted to anyhow::Error and displayed).
   - `handle_gate_history()` line ~598: same pattern
   - `handle_gate_history_detail()` line ~722: same pattern
   - `GateCommand::History { last: true, .. }` arm line ~121: same pattern

   **Pattern change:** Currently each site does:
   ```rust
   let entry = match assay_core::spec::load_spec_entry(name, &specs_dir) {
       Ok(e) => e,
       Err(assay_core::AssayError::SpecNotFound { .. }) => {
           bail!("spec '{name}' not found in {}", config.specs_dir);
       }
       Err(e) => return Err(e.into()),
   };
   ```
   Replace with:
   ```rust
   let entry = assay_core::spec::load_spec_entry_with_diagnostics(name, &specs_dir)?;
   ```
   The `SpecNotFoundDiagnostic` Display impl (from Plan 01) produces the full diagnostic message, and `anyhow` will use it.

3. **spec.rs** — `handle_spec_show()` line ~56: same pattern change.

**MCP call site:**

4. **server.rs** — `load_spec_entry_mcp()` at line ~1075: currently calls `assay_core::spec::load_spec_entry(name, &specs_dir).map_err(|e| domain_error(&e))`. Change to `assay_core::spec::load_spec_entry_with_diagnostics(name, &specs_dir).map_err(|e| domain_error(&e))`. The `SpecNotFoundDiagnostic` Display will be used by `domain_error()` which calls `err.to_string()`, so the enriched message automatically flows to MCP consumers.

1. **config/mod.rs** `load()` line ~88:
   ```rust
   // Before:
   let config: Config = toml::from_str(&content).map_err(|e| AssayError::ConfigParse {
       path: path.clone(),
       message: e.to_string(),
   })?;
   // After:
   let config: Config = toml::from_str(&content).map_err(|e| AssayError::ConfigParse {
       path: path.clone(),

   })?;
   ```

2. **spec/mod.rs** `load()` line ~211:
   ```rust
   // Before:
   let spec: Spec = toml::from_str(&content).map_err(|e| AssayError::SpecParse {

   // After:
   let spec: Spec = toml::from_str(&content).map_err(|e| AssayError::SpecParse {
       path: path.to_path_buf(),

   })?;
   ```

   Same pattern — replace `e.to_string()` with `crate::config::format_toml_error(&content, &e)` in the `GatesSpecParse` map_err.
