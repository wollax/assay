<feature>
  <name>ERR-01: Command error classification and formatting</name>
  <files>crates/assay-core/src/gate/mod.rs</files>
  <behavior>
    Pure functions for exit code classification and error message formatting.
    Commands spawn via `sh -c`, so:
    - Exit code 127 = user's binary not found (common case)
    - Exit code 126 = user's binary not executable (common case)
    - io::ErrorKind::NotFound from .spawn() = sh itself missing (near-impossible, still handle)

    Cases:
    - extract_binary("cargo test --release") -> "cargo"
    - extract_binary("sh") -> "sh"
    - extract_binary("") -> ""
    - classify_exit_code(127) -> Some(CommandNotFound)
    - classify_exit_code(126) -> Some(PermissionDenied)
    - classify_exit_code(1) -> None
    - classify_exit_code(0) -> None
    - format_command_error for exit 127 with cmd "cargo test" -> contains "cargo" and "not found" and "PATH"
    - format_command_error for exit 126 with cmd "cargo test" -> contains "cargo" and "not executable" and "permissions"
    - format_command_error for spawn NotFound (sh missing) -> contains "sh" and "not found"
    - format_command_error for spawn PermissionDenied -> contains "not executable"
  </behavior>
  <implementation>
    Add to `crates/assay-core/src/gate/mod.rs`:

    1. `pub(crate) fn extract_binary(cmd: &str) -> &str` — returns first whitespace-delimited token, or empty string.

    2. An enum or simple classification:
    ```rust
    pub(crate) enum CommandErrorKind {
        NotFound,
        NotExecutable,
    }
    pub(crate) fn classify_exit_code(code: i32) -> Option<CommandErrorKind> { ... }
    ```

    3. `pub(crate) fn format_command_error(cmd: &str, kind: CommandErrorKind) -> String` — produces the user-facing message. For NotFound: `"command '{binary}' not found. Is it installed and in PATH?"`. For NotExecutable: `"command '{binary}' found but not executable. Check file permissions."`.

    Tests go in the existing `#[cfg(test)] mod tests` block in gate/mod.rs.
  </implementation>
</feature>

<feature>
  <name>ERR-02: Spec-not-found diagnostics</name>
  <files>crates/assay-core/src/spec/mod.rs, crates/assay-core/src/error.rs</files>
  <behavior>
    Levenshtein distance, fuzzy matching, and diagnostic error formatting.

    Levenshtein cases:
    - levenshtein("", "") -> 0
    - levenshtein("", "abc") -> 3
    - levenshtein("abc", "") -> 3
    - levenshtein("kitten", "sitting") -> 3
    - levenshtein("same", "same") -> 0
    - levenshtein("a", "b") -> 1

    Fuzzy match cases:
    - find_fuzzy_match("auth-flw", &["auth-flow", "billing", "login"]) -> Some("auth-flow") (distance 1)
    - find_fuzzy_match("xyz", &["auth-flow", "billing"]) -> None (too far)
    - find_fuzzy_match("ab", &["ac", "ad"]) -> None (two matches within threshold = ambiguous)
    - find_fuzzy_match("a", &["b"]) -> None (distance 1 but 1 > len/2=0, threshold blocks short names)
    - find_fuzzy_match("test", &[]) -> None

    Diagnostic format cases (format_spec_not_found):
    - Zero specs, no invalid: "No specs found in .assay/specs/."
    - 3 specs: "spec 'xyz' not found. Available specs: alpha, beta, gamma"
    - 11 specs: "spec 'xyz' not found. Available specs: a, b, c, d, e, f, g, h, i, j (and 1 more)"
    - With invalid: includes "billing (invalid)" in list
    - With fuzzy match: appends "\nDid you mean 'auth-flow'?"
    - Invalid specs also get separate warning line
  </behavior>
  <implementation>
    1. Add to `crates/assay-core/src/spec/mod.rs`:
    - `fn levenshtein(a: &str, b: &str) -> usize` — standard DP algorithm with 1D vector. Keep private (pub(crate) for testing).
    - `fn find_fuzzy_match(name: &str, candidates: &[String]) -> Option<String>` — iterate candidates, compute distance, return Some only if exactly one candidate has distance <= 2 AND distance <= name.len() / 2.
    - `pub fn format_spec_not_found(name: &str, specs_dir: &Path, available: &[String], invalid: &[String], suggestion: Option<&str>) -> String` — produces the full diagnostic message.

    2. Add to `crates/assay-core/src/error.rs`:
    - `SpecNotFoundDiagnostic` variant:
    ```rust
    #[error("{}", format_spec_not_found_display(.name, .specs_dir, .available, .invalid, .suggestion.as_deref()))]
    SpecNotFoundDiagnostic {
        name: String,
        specs_dir: PathBuf,
        available: Vec<String>,
        invalid: Vec<String>,
        suggestion: Option<String>,
    }
    ```
    The `format_spec_not_found_display` function is a free function in error.rs that delegates to the formatting logic (or is self-contained). The actual formatting logic can live in spec/mod.rs and be called from error.rs, or be duplicated minimally.

    **Design choice:** Put the formatting function in `spec/mod.rs` as `pub(crate) fn format_spec_not_found(...)` and call it from the Display impl helper in error.rs. Alternatively, make it a standalone function in error.rs to avoid circular dependency concerns. Since error.rs doesn't import spec, put the format function in error.rs directly.

    Tests in spec/mod.rs for levenshtein and find_fuzzy_match. Tests in error.rs for SpecNotFoundDiagnostic Display.
  </implementation>
</feature>

<feature>
  <name>ERR-03: TOML parse error formatting</name>
  <files>crates/assay-core/src/config/mod.rs</files>
  <behavior>
    Custom TOML error formatting and source line truncation.

    Source line truncation cases:
    - line "short" with budget 80 -> passthrough, caret_offset = col
    - line of 120 chars, col=60, budget=80 -> center around col, ellipsis on both sides
    - line of 120 chars, col=5, budget=80 -> no left ellipsis, right ellipsis
    - line of 120 chars, col=115, budget=80 -> left ellipsis, no right ellipsis

    format_toml_error cases:
    - With span: shows "line N, column M: {message}" + source line + caret
    - Without span: falls back to just the error message
    - With long source line: truncates around error column
    - File path is shown by the caller (the AssayError Display impl already prepends "parsing config '{path}': {message}")

    The `toml::de::Error::to_string()` already includes rich formatting. HOWEVER, our custom formatter gives us control over:
    - Source line truncation (toml crate doesn't truncate)
    - Consistent formatting across all parse error types
    - Integration with our file-path-first Display format

    **Key insight from research:** `toml::de::Error` has `.span()` -> `Option<Range<usize>>` and `.message()` -> `&str`. Use these to build custom output. Also need `translate_position(content, byte_offset) -> (line, col)` to convert span start to line/column.
  </behavior>
  <implementation>
    Add to a suitable location (recommend `crates/assay-core/src/config/mod.rs` since it's shared TOML parsing logic, or create a small `toml_fmt` helper module — but keeping it in config/mod.rs is simpler since spec/mod.rs can import it):

    Actually, since both config/mod.rs and spec/mod.rs need it, put it in a new private module or as pub(crate) functions accessible from both. Best location: add functions directly to `crates/assay-core/src/config/mod.rs` and make them `pub(crate)` so spec/mod.rs can call them. Or add to error.rs. **Decision:** Put in config/mod.rs as `pub(crate)` since config is already a dependency of nothing within assay-core (spec/mod.rs and config/mod.rs are siblings). Actually they're both in assay-core so they can call each other's pub(crate) items via `crate::config::format_toml_error`.

    1. `pub(crate) fn translate_position(content: &str, byte_offset: usize) -> (usize, usize)` — count newlines before byte_offset for line, chars since last newline for column.

    2. `pub(crate) struct TruncatedLine { pub text: String, pub caret_offset: usize }`

    3. `pub(crate) fn truncate_source_line(line: &str, col: usize, budget: usize) -> TruncatedLine` — if line fits in budget, passthrough. Otherwise center around col with `...` ellipsis markers.

    4. `pub(crate) fn format_toml_error(content: &str, err: &toml::de::Error) -> String` — uses span(), message(), translate_position, truncate_source_line to produce:
    ```
    line 3, column 10: Unexpected `x`
      |
    3 | some_key = x_bad_value
      |            ^
    ```
    Falls back to `err.message().to_string()` when span is None.

    Note: The file path is NOT included in format_toml_error output — it's already handled by the AssayError Display impl (`"parsing config '{path}': {message}"`). The format_toml_error produces the `{message}` part.

    Tests in config/mod.rs tests module.
  </implementation>
</feature>
