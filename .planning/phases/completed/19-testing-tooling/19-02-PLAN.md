**Write direct handler tests in server.rs `#[cfg(test)] mod tests`:**

Add async handler tests that construct `AssayServer::new()`, set up tempdir with `create_project()` and `create_spec()`, call `std::env::set_current_dir(dir.path())`, then call handler methods directly.

Required tests (minimum, add more if natural):

1. `spec_list_valid_project_returns_specs` — create project + spec, call `server.spec_list().await`, verify response contains the spec name and criteria count
2. `spec_get_valid_spec_returns_content` — create project + spec, call `server.spec_get(Parameters(SpecGetParams { name })).await`, verify response contains spec TOML content
3. `spec_get_missing_spec_returns_error` — call spec_get for nonexistent spec, verify `is_error` is true
4. `gate_run_command_spec_returns_results` — create project + spec with `cmd = "echo ok"` criteria, call `server.gate_run(Parameters(GateRunParams { ... })).await`, verify passed count > 0
5. `gate_run_nonexistent_spec_returns_error` — call gate_run for nonexistent spec, verify error response
6. `gate_run_nonexistent_working_dir_returns_error` — call gate_run with invalid working_dir, verify error about path not existing
7. `gate_report_invalid_session_returns_error` — call gate_report with a fake session_id, verify error
8. `gate_history_no_history_returns_empty` — call gate_history for spec with no history, verify empty runs list

For each test:
- Use `#[tokio::test]` attribute
- Follow naming convention: `{feature}_{scenario}_{expected}`
- Use `extract_text()` helper to get response text
- Parse response JSON with `serde_json::from_str`
- Add `insta::assert_json_snapshot!` for at least 2 representative responses (spec_list and gate_run happy paths)

**CWD race condition mitigation:** Since these tests use `set_current_dir`, they will conflict if run in parallel. To handle this without adding serial_test:
- Keep CWD-dependent handler tests minimal in the unit test module
- Put the bulk of CWD-dependent tests in the integration test file (task 2), which runs in a separate process

Note: `spec_list()` takes no parameters. `spec_get()`, `gate_run()`, `gate_report()`, `gate_finalize()`, and `gate_history()` take `Parameters(T)` wrappers. Construct as `Parameters(SpecGetParams { name: "...".to_string() })` etc.

**Part A: Integration tests for MCP handler lifecycle**

Create `crates/assay-mcp/tests/mcp_handlers.rs` with integration tests. Each test in this file runs in its own process, avoiding CWD races.

Required integration tests:

1. `gate_lifecycle_run_report_finalize` — Full agent gate lifecycle:
   - Create project with mixed spec (cmd + AgentReport criteria)
   - Call gate_run → extract session_id from response
   - Call gate_report with session_id, criterion name, pass/fail, reasoning
   - Call gate_finalize with session_id
   - Verify finalize response indicates persisted=true
   - Verify history file exists in `.assay/results/` directory

2. `gate_run_with_timeout` — Create project + spec with `cmd = "echo ok"`, call gate_run with timeout=10, verify results returned within timeout

3. `spec_list_with_parse_errors` — Create project with one valid spec and one malformed TOML file in specs/, call spec_list, verify response contains both a spec entry AND an error entry

Use the same tempdir helpers pattern from server.rs tests. Import `assay_mcp::AssayServer` and the parameter types. Add `insta::assert_json_snapshot!` for the lifecycle test's finalize response.

**Part B: Open issue triage and test gap resolution**

1. Read ALL 25 test-related issue files in `.planning/issues/open/`
2. For each issue:
   - Check if the scenario is already tested by existing code (cross-reference with `grep` for test function names)
   - If already covered: move to `.planning/issues/closed/` with a note
   - If still valid: implement the test in the appropriate crate's test module
   - If stale (invalidated by later refactors): move to `.planning/issues/closed/` with explanation

Key test gaps to address (from Phase 3/6 reviews):
- GateKind unknown variant deserialization test (assay-types)
- GateResult JSON roundtrip (assay-types)
- Criterion deser failure test (assay-types)

- Enforcement TOML roundtrip (assay-types)
- Mixed enforcement failures (assay-core/gate)
- History load error path tests (assay-core/history)

Add these tests to the `#[cfg(test)]` modules of their respective source files, following the `{feature}_{scenario}_{expected}` naming convention.

3. Run `cargo test --workspace` to confirm all tests pass
4. Create `.planning/issues/closed/` directory if it doesn't exist
5. Move resolved issue files with `git mv` from open/ to closed/
