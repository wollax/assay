<name>Task 1: Add raw_line to ParsedEntry and pruning types to assay-types</name>
<files>
crates/assay-core/src/context/parser.rs

</files>
<action>
1. In `crates/assay-core/src/context/parser.rs`, add a `raw_line: String` field to `ParsedEntry`:
   ```rust
   pub struct ParsedEntry {
       pub entry: SessionEntry,
       pub line_number: usize,
       pub raw_bytes: usize,
       /// The original JSON line, preserved for lossless write-back.
       pub raw_line: String,
   }
   ```
   Update `parse_session` to clone the line into `raw_line` before deserializing. Update all test helper `ParsedEntry` constructions (in parser.rs tests and anywhere else that constructs ParsedEntry directly) to include `raw_line`.

2. Add a helper method to ParsedEntry for content modification:
   ```rust
   impl ParsedEntry {
       /// Re-serialize the entry after modification, updating raw_line and raw_bytes.
       pub fn update_content(&mut self, new_entry: SessionEntry) {
           self.raw_line = serde_json::to_string(&new_entry).unwrap_or_default();
           self.raw_bytes = self.raw_line.len();
           self.entry = new_entry;
       }
   }
   ```

3. In `crates/assay-types/src/context.rs`, add these types at the bottom (before the last inventory::submit block or after it):
   - `PruneStrategy` enum with variants: ProgressCollapse, SystemReminderDedup, MetadataStrip, StaleReads, ThinkingBlocks, ToolOutputTrim. Derive Debug, Clone, Copy, PartialEq, Eq, Hash, Serialize, Deserialize, JsonSchema. Use `#[serde(rename_all = "kebab-case")]`.
   - `PrescriptionTier` enum with variants: Gentle, Standard, Aggressive. Same derives + `#[serde(rename_all = "kebab-case")]`.
   - Implement `PrescriptionTier::strategies(&self) -> &[PruneStrategy]` returning the correct strategy subsets in execution order (line-deletion first, then content-modification). See RESEARCH.md for the exact mapping.
   - `PruneSummary` struct: strategy (PruneStrategy), lines_removed (usize), lines_modified (usize), bytes_saved (u64), protected_skipped (usize), samples (Vec of PruneSample).
   - `PruneSample` struct: line_number (usize), description (String), bytes (u64).
   - `PruneReport` struct: session_id (String), original_size (u64), final_size (u64), original_entries (usize), final_entries (usize), strategies (Vec of PruneSummary), executed (bool).
   - Register PruneReport with inventory::submit for JSON schema.

4. Search all files that construct `ParsedEntry` directly (e.g., checkpoint/extractor.rs tests) and add the `raw_line` field. Use an empty string or a reasonable default for test fixtures since those entries aren't written back.
</action>

<name>Task 3: Implement team message protection set</name>
<files>
crates/assay-core/src/context/pruning/protection.rs

<action>
1. Create `crates/assay-core/src/context/pruning/protection.rs` with:

2. Define `PROTECTED_TOOL_NAMES` constant array:
   ```rust
   const PROTECTED_TOOL_NAMES: &[&str] = &[
       "Task", "TaskCreate", "TaskUpdate", "TaskOutput", "TaskGet", "TaskList", "TaskStop",
       "TeamCreate", "TeamDelete",
       "SendMessage",
   ];
   ```

3. Implement `pub fn build_protection_set(entries: &[ParsedEntry]) -> HashSet<usize>`:
   - Iterate all entries
   - For each entry, call `is_team_message(entry)`
   - If true, insert `entry.line_number` into the set
   - Return the set

4. Implement `fn is_team_message(entry: &ParsedEntry) -> bool`:
   - For `SessionEntry::User(u)`: check if `u.message` is an array of blocks, and if any block has `name` matching PROTECTED_TOOL_NAMES (same pattern as checkpoint/extractor.rs)
   - For `SessionEntry::Progress(p)`: check nested `data.message.message.content[]` blocks for tool_use with protected names (same pattern as extractor.rs extract_tasks)
   - For `SessionEntry::Assistant(a)`: check content blocks for ToolUse with protected names
   - All other variants: return false

5. Write comprehensive tests:
   - Test with user entry containing TaskCreate tool_use -> protected
   - Test with progress entry containing SendMessage -> protected
   - Test with assistant entry containing TeamCreate tool_use -> protected
   - Test with regular user message -> not protected
   - Test with progress tick (no tool_use) -> not protected
   - Test that protection set contains correct line numbers
   - Test empty entries -> empty protection set
</action>
